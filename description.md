# 实验 2：一般网络与可修系统可靠性分析

## 一、实验目标

本实验以桥式起重机机械系统为背景，在同一组工程对象上完成两类可靠性分析：

1. 一般网络系统可靠性分析
   将原本可以用 RBD 表示的系统，进一步抽象为一般网络结构，分析系统在给定任务时间内完成任务的可靠度。

2. 可修系统可靠性分析
   在元件失效率基础上引入维修率，分析系统的稳态可用度，并比较不可修与可修模型的差异。

通过本实验，你应掌握：

- 如何从工程系统中提取成功事件与系统边界
- 如何将串并联结构扩展为一般网络模型
- 如何根据任务剖面计算元件占空比与有效失效率
- 如何计算单元件可靠度、系统可靠度和系统可用度
- 如何解释薄弱环节，并提出工程改进建议

---

## 二、工程背景

某自动化桥式起重机承担“工位 A 到工位 B”的托盘搬运任务。一个完整搬运循环由以下 5 个阶段组成：

1. Pick：下降到位并完成夹具闭合确认
2. Lift：起升到安全高度并确认上限位有效
3. TravelLoaded：载荷行走到目标工位并保持位置反馈有效
4. Place：下降到位并完成夹具打开确认
5. ReturnEmpty：空载返回起始工位并保持位置反馈有效

设：

- 单循环时长为 `t_cyc`
- 班次循环次数为 `N`
- 总任务时间为 `T = N * t_cyc`

本实验仍然保留以上机械系统背景，但分析对象不再局限于简单串并联系统，而是扩展为一般网络和可修系统。

---

## 三、任务成功事件与建模口径

### 1. 一般网络模型

系统被抽象为一个从 `source` 到 `target` 的功能网络。

成功事件定义为：

- 在任务时间 `T` 内，网络中存在至少一条从源点到汇点的可用功能路径
- 该路径能够支撑完整搬运循环所需的供电、控制、执行、反馈与夹具动作

### 2. 可修系统模型

在一般网络模型基础上，为每个关键元件引入维修率 `mu`。

分析目标为：

- 单元件稳态可用度
- 系统稳态可用度
- 维修能力变化对系统性能的影响

### 3. 统一假设

本实验默认采用以下简化假设：

- 元件失效相互独立
- 元件失效服从指数分布
- 元件维修服从指数分布
- 对阶段性工作的元件，采用占空比折算有效失效率
- 网络边的可用性由对应元件状态决定

## 四、实验数据文件说明

实验所需输入数据位于 `data/` 目录中。

### 1. `data/components.csv`

该文件给出元件基础参数，包括：

- `id`：元件编号
- `name`：元件名称
- `lambda_per_h`：失效率，单位 `1/h`
- `mu_per_h`：维修率，单位 `1/h`

说明：

- `lambda_per_h` 用于不可修系统的可靠度计算
- `mu_per_h` 用于可修系统的稳态可用度计算

### 2. `data/mission_profile.csv`

该文件给出任务剖面。

每一行对应一个任务阶段，至少包含：

- `phase`
- `duration_h`
- 各元件在该阶段是否工作，取值为 `0` 或 `1`

作用：

- 计算单循环时长 `t_cyc`
- 计算各元件占空比 `duty`
- 进一步计算有效失效率 `lambda_eff`

占空比定义为：

`duty = 元件工作总时长 / 单循环总时长`

### 3. `data/model.json`

该文件给出系统结构模型，主要包含：

- `network_model`
- `repairable_model`

其中：

`network_model` 用于描述一般网络结构，包含：

- `source`：源点
- `target`：汇点
- `nodes`：网络节点
- `edges`：网络边

每条边通常包括：

- `id`
- `from`
- `to`
- `component`
- `role`

`repairable_model` 用于描述可修系统分析口径，当前采用稳态可用度模型。

### 4. 机械部件到网络边的映射示例（建议参考）

可按“部件承担的功能连接”来映射网络边，例如：

- `C1 断路器`：`START -> PWR_IN`
- `C6 PLC控制器`：`CTRL_CORE -> SAFETY_OK`
- `C8 起升变频器`：`ACT_GATE -> HOIST_PATH`
- `C12 小车变频器`：`ACT_GATE -> TROLLEY_PATH`
- `C19 夹具执行器`：`POSITION_OK -> TASK_DONE`

上面的例子表示：从电源、控制、执行到任务完成的功能链路。你可以在此基础上加入冗余边或桥接边，但必须保证含义清晰。

### 5. `model.json` 字段规则（必须/可选）

必须字段：

- 顶层：`network_model`、`repairable_model`
- `network_model`：`source`、`target`、`nodes`、`edges`
- 每条 `edge`：`id`、`from`、`to`、`component`

可选字段：

- 顶层：`description`、`hints`、`model`
- 每条 `edge`：`role`
- `repairable_model`：`metric`、`component_repair_field`、`use_effective_lambda`、`notes`

最小合法样例（仅用于说明格式）：

```json
{
  "network_model": {
    "source": "START",
    "target": "TASK_DONE",
    "nodes": ["START", "TASK_DONE"],
    "edges": [
      {
        "id": "E1",
        "from": "START",
        "to": "TASK_DONE",
        "component": "C1"
      }
    ]
  },
  "repairable_model": {
    "use_effective_lambda": true
  }
}
```

## 五、你需要完成的任务

### 任务 1：理解系统边界并说明成功事件

你需要在报告中说明：

- 为什么选择当前的 `source` 与 `target`
- 哪些元件属于任务成功所必需的功能路径
- 哪些元件体现了冗余、桥接或可替代路径
- 一般网络模型与原始 RBD 思路之间的关系

### 任务 2：完成一般网络可靠性计算

你需要：

1. 根据任务剖面计算 `t_cyc`
2. 根据给定 `N` 计算总任务时间 `T`
3. 对每个元件计算：
   - `duty`
   - `lambda_eff = lambda * duty`
   - `R_i(T) = exp(-lambda_eff * T)`
4. 根据 `network_model` 计算系统任务可靠度 `R_sys(T)`

最低核验要求：

- 至少手算 1 个元件的 `R_i(T)`（写出代入过程）
- 至少手算 1 条局部路径或局部子结构的可靠度
- 将手算结果与脚本输出对比，并说明误差来源（如四舍五入）

### 任务 3：完成可修系统可用度计算

你需要：

1. 读取每个元件的维修率 `mu`
2. 计算单元件稳态可用度
3. 根据系统网络结构计算系统稳态可用度 `A_sys`
4. 比较 `R_sys(T)` 与 `A_sys` 的差异，并解释原因

口径要求：

- 本实验默认使用 `A_i = mu / (lambda_eff + mu)`，对应 `repairable_model.use_effective_lambda = true`
- 若你改为 `lambda` 口径，必须在报告中说明并解释差异

### 任务 4：完成结果解释与工程分析

你需要在报告中完成以下文字分析：

- 找出系统薄弱环节
- 解释最薄弱元件为什么会成为限制因素
- 比较可靠度模型与可用度模型下的差异
- 提出至少 1 条工程改进建议，并说明预期影响方向

最低完成标准（未达到视为任务4未完成）：

- 至少 3 段独立分析文字（每段聚焦一个问题）
- 必须包含“为什么薄弱”而不是只写“哪个最薄弱”
- 改进建议必须写明预期影响方向（提升可靠度、提升可用度或同时提升）

## 六、主要公式

### 1. 总任务时间

`T = N * t_cyc`

### 2. 有效失效率

`lambda_eff = lambda * duty`

### 3. 单元件任务可靠度

`R_i(T) = exp(-lambda_eff * T)`

### 4. 单元件稳态可用度

在指数失效、指数维修假设下：

`A_i = mu / (lambda + mu)`

如果实验口径要求将占空比折算到可修系统中，则可使用：

`A_i = mu / (lambda_eff + mu)`

本仓库默认采用 `lambda_eff` 口径（`use_effective_lambda = true`）。

示例（1 行）：

- 若 `lambda_eff = 2.0e-5 1/h`，`mu = 0.5 1/h`，则 `A_i = 0.5 / (0.50002) ≈ 0.99996`

### 5. 系统层指标

- 一般网络系统可靠度：由网络中从源点到汇点的连通概率确定
- 可修系统可用度：由网络中从源点到汇点的可用连通概率确定

注意：

- 不要将 `duty` 直接乘到可靠度上
- 不要将一般网络简单等同为一个单一串联或并联系统
- 必须先定义成功事件，再讨论系统可靠度或可用度

## 九、sanity checks 说明

程序内置若干合理性检查，用于帮助你验证模型方向是否正确。

### 1. 去冗余应变差

将部分冗余边去除后：

- 系统可靠度不应升高
- 系统可用度不应升高

### 2. 缩短任务时间应变好

将任务时间减半后：

- 系统可靠度应提高

### 3. 降低维修率应变差

将维修率降低后：

- 系统可用度应下降

如果以上规律不满足，通常说明模型结构、边界或参数使用存在问题。

## 十二、常见错误对照表

| 常见错误 | 典型表现 | 修正建议 |
|---|---|---|
| 将 `duty` 直接乘到可靠度上 | 写成 `R = duty * exp(-lambda*T)` | 先算 `lambda_eff = lambda*duty`，再算 `R_i(T)` |
| 混淆可靠度与可用度 | 把 `R(t)` 与 `A` 当成同一指标 | 在报告中单列两组定义与物理意义 |
| 没有定义成功事件 | 直接套公式但说不清 source/target 含义 | 先写成功事件，再开始计算 |
| 一般网络退化成串并联 | 忽略桥接边或共享节点 | 在网络图中标出跨支路连接并解释作用 |
| 单位不统一 | `lambda` 用 `1/h`，`T` 直接用分钟 | 全部换算到小时后再代入 |
| 只给数字不解释 | 有结果表格但无工程含义 | 任务4至少完成 3 段解释并给出改进方向 |

最终报告不应只停留在“给出一个数字”，而应回答以下问题：

- 系统在任务时间内完成任务的可能性有多大？
